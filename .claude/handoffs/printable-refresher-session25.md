## Gotchas / Context

1. **Per-project threshold tuning is the right knob** — chromium stays at the strict default; firefox + webkit get relaxed defaults. This keeps chromium as the gold-standard regression detector while accommodating genuine cross-browser jitter without false positives. A real layout regression that slips past firefox/webkit but is caught by chromium is the safety net working as designed.

2. **Webkit on Windows is a Playwright build, not Safari proper** — font availability differs from real Safari (no Georgia; system serif fallback). Tests run + assert layout equivalence; a real "Safari user" test would need a macOS or Linux runner. Acceptable for visual regression of the layout/CSS contract; not a substitute for real-Safari user testing.

3. **Wall time grew 8s → 7.1m** — webServer spawns once per browser project (3 spawns × ~1m boot each). The dev server itself starts fast (~400ms); the spawn overhead is the Vite plugin init + first-render warm-up. Acceptable for local + manual CI runs; parallelization to 3 workers would cut to ~2.5m at the cost of dev-server-per-worker isolation complexity.

4. **All 21 baselines stable on first try** — surprising given the cross-browser jitter; per-project thresholds were tuned conservatively + the actual diffs were within tolerance. Future sessions might need to tighten if false-negatives surface.

5. **No script changes for S25** — `npm run test:visual` / `:update` / `:debug` (added in S23) work identically with the new browsers. `playwright test` automatically runs all 3 projects; `playwright test --project=chromium` runs just one.

6. **Browser binaries are NOT in npm** — `~/.cache/ms-playwright/` (Linux/Mac) or `%LOCALAPPDATA%\ms-playwright` (Windows). Each developer machine + CI runner needs `npx playwright install firefox webkit` once. Not committed; not in package-lock.

7. **firefox + webkit baselines ARE tracked content** — the existing `!tests/playwright/**/*.png` exception in `.gitignore` (from S23) covers all baselines regardless of browser. No gitignore changes needed.

8. **The 8th test (e2e behavior-only) ran 3 times** — once per browser project. Each browser confirms the Settings → Refresher → banner-appears flow independently. This is a good sanity check that the IDB seeding + hash-deep-link navigation works cross-browser.

9. **Webkit's first run has higher wall time per test** (3.5-4.5s vs chromium's 1.0-1.5s) due to webkit's slower first-paint on Windows. Subsequent test runs in the same pass are faster (subsequent ~3s) but never as fast as chromium. Trade-off accepted for cross-browser coverage.
