# Concurrent Session Safety

CWOS supports multiple simultaneous Claude sessions (e.g., two terminals, or a background agent running while you work interactively).

## How It Works

### Session Isolation
- Each session registers in `.active-sessions/{session-id}.lock`
- Sessions can see each other but operate independently
- Session-end cleans up the lock file

### Counter Safety
- IDs (WS-NNN, FIND-NNN, etc.) are **scan-derived, not stored**
- To create a new item: scan the directory for the highest existing ID, use max+1
- The `config.yaml` counters are advisory cache — never authoritative
- This eliminates race conditions between concurrent sessions

### Work Item Claim Protection
- When claiming a work item (`/next` or `/workstream claim`), check the item's `claimed_by` field first
- If already claimed by another session: skip it, try the next item
- Claims are written atomically to the item's YAML file
- A second session scanning the same item will see it already claimed

### State Updates
- `system/state.md` uses last-writer-wins (acceptable for a solo user)
- `system-summary.yaml` is regenerated by session-end and status
- Program YAML files are updated individually (low collision risk)

## When Conflicts Can Still Occur
- Two sessions editing the same work item YAML simultaneously (rare — claim checking prevents this)
- Two sessions updating state.md at the exact same moment (harmless — the next session re-reads)
- Background agents creating findings while you're creating items (IDs won't collide due to scan-derived strategy)
