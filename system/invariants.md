The load-bearing primitive for downstream river-decision code (Basin, future bluff-catch / thin-value sizing) is the **showdown-outcome distribution** with strength-tier bucketing of villain's combos against the public board — not a variance-across-runouts metric. Buckets are hero-agnostic and range-agnostic — they describe the combo's current 5-card strength vs all legal combos on the same board.

| ID | Invariant | Why it matters | How to check | Verified |
|----|-----------|----------------|--------------|----------|
| INV-STRENGTH-BUCKET-1 | At every postflop street (flop / turn / river), every entry in `perCombo` from `computeComboEquityDistribution` carries a `villainStrengthBucket ∈ STRENGTH_BUCKETS = ['nuts', 'very-strong', 'strong', 'marginal', 'weak-or-bluff']`. The field is `null` only when the combo collides with a board card (skipped earlier by the segCombo filter), the combo has `card1 === card2`, or the board falls outside the 3..5 card range. | Downstream consumers (Basin classifier, future river surfaces) read this field to compose strength-tier histograms over a range. A missing or `undefined` field forces every consumer to add defensive guards — the explicit five-tier enumeration is the contract. | `gameTreeEquity.test.js` `'villainStrengthBucket (WS-194)'` block: asserts every `perCombo` entry has the field and the value is in STRENGTH_BUCKETS on flop / turn / river. `handEvaluator.test.js` `'computeBoardStrengthTable'` block: asserts the table populates every legal combo on the board. | 2026-05-19 |
| INV-STRENGTH-BUCKET-2 | Bucketing is **rank-relative per board**: each combo's bucket is determined by its score's percentile in the full enumeration of legal 2-card combos on that board. Bands are `[≤5% nuts, ≤20% very-strong, ≤50% strong, ≤80% marginal, ≤100% weak-or-bluff]`. Tied scores share the bucket of the group's first member (ties on a boundary settle into the higher tier). | Calibrated absolute thresholds (e.g., "set or better → nuts") would mis-classify on textured boards — top-pair-top-kicker is a top-tier hand on K72r but a marginal hand on Ah7h2h. Tie-handling-by-first-member keeps the algorithm deterministic and avoids percentile-rank flapping at boundaries. | `handEvaluator.test.js` board-adaptive test (`KK lands in 'nuts' on K72r but not on Ah7h2h`); ties-share-bucket test on a board with deterministic ties. | 2026-05-19 |
| INV-STRENGTH-BUCKET-3 | `villainStrengthBucket` is computed against the **public board only** — never against hero's hand, hero's equity, or villain's range. The bucket primitive is shared infrastructure — making it reusable from `rangeEngine` or `shapeLanguage` later requires the one-way dependency rule to hold. | Cross-domain grep guard: `grep -rn "from '../exploitEngine\\|from '../rangeEngine\\|from '../shapeLanguage'" src/utils/pokerCore/` returns no matches. | 2026-05-19 |

## INV-RL-DETERMINISM family (2026-05-20, SPR-095 / WS-205)

Range Lab's per-street range evolution (DS-54) re-derives per-combo equity on every turn/river card. The equity-precompute cache (`src/utils/rangeEngine/equityCache.js`) makes this fast, but the speed must never come from sampling — a cached equity that depended on RNG state would make the same spot show different numbers on re-entry, and would defeat the LSW↔RL parity invariant (WS-206). All Range Lab equity is exact exhaustive enumeration.

| ID | Invariant | Why it matters | How to check | Verified |
|----|-----------|----------------|--------------|----------|
| INV-RL-DETERMINISM-1 | All Range Lab per-combo equity is computed by `exactComboEquity` (`src/utils/pokerCore/monteCarloEquity.js`) — pure exhaustive runout enumeration over a 3–5 card board, no `Math.random`. The same `(heroCards, villainCombo, board)` always yields the bit-identical equity. | A Range Lab spot must surface the same equity every time it is entered; a sampled value would flap between visits and could not be validated against authored LSW content (WS-206). Exact enumeration is also tractable here (flop = C(47,2)=1081 runouts/combo, computed once at flop-paint-finalize and cached). | `monteCarloEquity.test.js` `'exactComboEquity'` determinism + symmetry tests; `equityCache.test.js` `'INV-RL-DETERMINISM'` block (two independent precomputes are bit-identical, NaN-aware). Grep guard: `grep -n "Math.random" src/utils/rangeEngine/equityCache.js` returns no matches. | 2026-05-20 |
| INV-RL-DETERMINISM-2 | The equity cache lives in `rangeEngine/` and imports its equity kernel **only** from `pokerCore` (`exactComboEquity`) — never from `exploitEngine/`. There is one source of per-combo equity truth, shared with `INV-LSW-RL-EQUITY-PARITY` (WS-206). | A second equity implementation inside the cache would (a) invert the intended `rangeEngine → pokerCore` dependency direction if it reached into `exploitEngine/`, and (b) silently drift from the engine, making WS-206's parity test validate against the wrong baseline. | Grep guard: `grep -rn "from '../exploitEngine" src/utils/rangeEngine/equityCache.js` returns no matches; the file imports `exactComboEquity` from `../pokerCore/monteCarloEquity`. | 2026-05-20 |
| INV-RL-DETERMINISM-3 | The cache is in-memory and session-scoped (a plain `Map`), cleared on app reload. It is never persisted to IndexedDB. | Persisting equity across reloads risks serving a stale value after an engine change altered the computation — a correctness hazard the cache must not introduce. Session-scoped TTL is the ratified design (SPR-095). | Grep guard: `equityCache.js` imports nothing from `src/utils/persistence/` and references no IDB API; storage is `new Map()`. | 2026-05-20 |

## INV-LSW-RL-EQUITY-PARITY family (2026-05-20, SPR-095 / WS-206)

Range Lab (RL) is the equity validator for Line Study (LSW) authored content (DS-55 JTBD). LSW line nodes carry the *inputs* for an equity computation (board, single-combo `heroView`, villain range via `villainRangeContext.baseRangeId`) but **no authored equity number** — the lines were written with qualitative reasoning. The invariant therefore pins the **engine's** computed hero-vs-range equity for each authored node to a committed, human-reviewed baseline (`lswRlParityBaseline.js`). It catches silent engine drift on authored teaching spots; it does NOT independently verify the original prose reasoning (nothing in the repo recorded numbers to verify against). **Reframed from the WS-206 charter's "match LSW asserted equity" after confirming no asserted-equity field exists** — founder-ratified 2026-05-20.

| ID | Invariant | Why it matters | How to check | Verified |
|----|-----------|----------------|--------------|----------|
| INV-LSW-RL-EQUITY-PARITY-1 | For every parity-eligible LSW node (3–5 card board + `heroView.kind==='single-combo'` + a `baseRangeId` that resolves via `villainRangeFor`), the engine's weighted hero-vs-range equity (sum of `exactComboEquity` over the resolved villain range) matches the committed baseline in `lswRlParityBaseline.js` within ±0.5%. | Without the pin, an engine change (rangeEngine / pokerCore / villain-range resolution) could silently move equity on an authored spot, breaking the lesson with no test failure. The pin forces a human to confirm any move is intended and the line still teaches correctly. ±0.5% = MC variance floor; exact enumeration → 0 drift in practice. | `src/utils/rangeEngine/__tests__/lswRlParity.test.js` `'engine equity for ... matches committed baseline'`. Regenerate baseline for review with `GEN_LSW_RL_BASELINE=1`. | 2026-05-20 |
| INV-LSW-RL-EQUITY-PARITY-2 | The parity equity uses pokerCore `exactComboEquity` — the same deterministic kernel as the Range Lab cache (`INV-RL-DETERMINISM-1`). There is no second equity path. | If the pin used a different equity implementation than RL, the two could diverge and the "RL validates LSW" contract would be meaningless. Shared kernel makes them equal by construction. | Test imports `exactComboEquity` from `../../pokerCore/monteCarloEquity`; no other equity function is called. | 2026-05-20 |
| INV-LSW-RL-EQUITY-PARITY-3 | Any eligible node whose inputs are illegal (hero combo collides with the board, or the villain range resolves empty) yields NaN equity and MUST appear in `KNOWN_DEGENERATE_NODES` with a reason. A new, undocumented degenerate node fails the test. | This is the LSW-content-validation half of the contract: RL flagging impossible authored holdings. It already caught `utg-vs-btn-4bp-deep::flop_root` declaring hero `A♥K♦` on a board containing `K♦` (impossible). Silent NaN-dropping would hide such content bugs. | `lswRlParity.test.js` `'every degenerate node is a documented known bug'`; the allowlist + reason live in `lswRlParityBaseline.js`. | 2026-05-20 |

**Failure-mode triage (which side is the bug):** a *baseline mismatch* means engine equity moved → inspect recent rangeEngine/pokerCore/villain-range changes, confirm correct, update the baseline value in the same commit with a note. A *new degenerate (NaN) node* means the LSW line node has illegal inputs (hero/board collision or empty range) → fix the line content (or document in `KNOWN_DEGENERATE_NODES` if intentional).
