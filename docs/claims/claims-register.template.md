## How to review

For each claim:

1. **Statement is accurate** — does the formal phrasing match what your product actually asserts?
2. **Source evidence is current** — does the cited file:line still contain the claim?
3. **Domain classification is correct** — clinical / legal / financial / scientific?
4. **Implicit vs explicit** — is the claim made directly, or only implied by UI affordances?
5. **Evidence tier is appropriate** — does the matrix in `evidence-standards-matrix.md` agree with the tier assigned here?
6. **Confidence is calibrated** — High/Medium/Low matches your read?

If a claim is wrong: edit, downgrade, or delete. If the engine missed a claim: add it (use the same field structure).

## Promotion checklist

Before renaming this file to `docs/claims/claims-register.md`:

- [ ] Every claim's formal statement reviewed against the source evidence
- [ ] Every claim's domain classification verified
- [ ] Every Low-confidence claim either upgraded, rewritten, or deleted
- [ ] Every Medium-confidence claim with non-empty review notes confirmed
- [ ] Every claim's `evidence_tier_required` cross-checked against `evidence-standards-matrix.draft.md`
- [ ] All "Claims NOT covered" gaps either filled in or accepted as deferred
- [ ] Each claim has a `last_verified` date set
- [ ] Scope-fence audit: every high-stakes domain claim (clinical/legal/financial) has a corresponding disclaimer at the cited surface

When checklist complete, rename and the register goes live. Drift detection (Phase 1) engages on subsequent baseline runs.
