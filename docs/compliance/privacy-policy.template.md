## How to review

For each section below:

1. **Statements are factually correct** — does each claim describe what your code actually does (not what it should do, what it does today)?
2. **Cited evidence is current** — do the cited file paths still implement what the section claims?
3. **Jurisdiction-specific clauses are valid** — does the language match the regulation's current requirements? (Especially: GDPR / CCPA / HIPAA wording is technical; defaults may be stale.)
4. **Confidence is calibrated** — High = repo-evidenced + jurisdiction-verified; Medium = inferred; Low = generic boilerplate flagged for your input.
5. **Founder-input markers are addressed** — sections marked `{{ FOUNDER MUST PROVIDE }}` are placeholders the engine could not infer.

If any section is wrong: edit, downgrade, or delete. The engine will not regenerate a section in a promoted file — your edits stick.
