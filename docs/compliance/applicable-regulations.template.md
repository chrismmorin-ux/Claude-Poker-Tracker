## How to review

For each regulation block:

1. **Applicability rationale is correct** — is the regulation actually triggered for your repo's customers and operations?
2. **Each requirement is load-bearing** — is the requirement something your application must DO (or NOT DO), or is it generic legal language?
3. **`enforced_by` paths are accurate** — does the cited code actually enforce the requirement?
4. **`status: missing` items are real gaps** — are they backlog candidates, or should the regulation actually not apply (in which case remove the requirement)?
5. **Founder-review notes addressed** — for any requirement with a non-empty review note, the founder must confirm before promotion.

## Promotion checklist

Before renaming this file to `docs/compliance/applicable-regulations.md`:

- [ ] Every regulation's applicability rationale verified against your customer base
- [ ] Every requirement is load-bearing (no generic legal language)
- [ ] Every `enforced_by` path verified (file, line, behavior)
- [ ] Every `status: missing` item triaged: either added to backlog OR regulation removed if it doesn't apply
- [ ] Every requirement has `last_verified` date set
- [ ] Cross-regulation conflicts resolved or escalated
- [ ] Legal counsel review (recommended for any jurisdiction outside founder's home country)

When checklist complete, rename and the regulation set goes live. Drift detection (Phase 1) will engage on subsequent baseline runs.
