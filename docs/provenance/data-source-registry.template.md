## How to review

For each source:

1. **Name & type are correct** — does the source name match what your code actually calls / reads from? Is `type` accurate (database vs external_api vs sdk_service vs fixture vs config_endpoint)?
2. **Source evidence is current** — does the cited file:line still implement the source declaration? Has the migration been squashed, the SDK dependency upgraded, the fixture moved?
3. **Surface class is right** — `internal_db` vs `derived` matters. A column populated from a vendor webhook is `derived`, not `internal_db`, even though it lives in your DB.
4. **Trust tier is calibrated** — does T1/T2/T3/T4/T5 match your read?
   - T1: Authoritative source with verifiable provenance (peer-reviewed, government registry, regulator API)
   - T2: Trusted vendor / canonical service (Stripe, AWS, Auth0, Twilio)
   - T3: Internal database with audit-logged write path
   - T4: Internal derivation / aggregation (computed from T1-T3)
   - T5: External unverified (scraped, third-party API without authority verification, user-submitted without validation)
5. **Confidence is calibrated** — High/Medium/Low matches your read?
6. **Review notes addressed** — for any non-empty review notes, the founder must confirm before promotion.

If a source is wrong: edit, downgrade, or delete. If the engine missed a source: add it under the appropriate `surface_class` section (use the same field structure).
