## How to review

For each data point:

1. **Chain is complete** — does every step from source to display appear, or did the engine skip a transformation? Common misses: SDK callbacks, background jobs, derived columns populated by webhooks.
2. **Trust tier monotonicity** — does the chain ever UPGRADE tier (e.g., T3 → T2)? **This is an inference error.** Tier never increases through the chain. A T3 source can stay T3 or downgrade to T4 (derivation), but it cannot become T2 by passing through your code.
3. **File:line citations are current** — does each step's cited code still implement the transform described?
4. **Transform descriptions accurate** — does "JSON parse" actually mean a parse, or is the engine eliding a schema-changing transformation?
5. **`tier_at_display` matches final step** — the displayed tier should equal the final chain step's `tier_at_step`.

If a chain is wrong: edit the steps, downgrade tier, or replace with `chain: []` and `review_notes: "founder will document manually"`. If the engine missed a data point: add it (use the same field structure).

---

## Trust-tier monotonicity (load-bearing)

Trust tier never increases through the chain. The engine's cross-check rejects any chain that violates this invariant. If you find an apparent upgrade in a draft chain that the cross-check missed, treat it as a defect and:

1. Verify the upstream source's actual tier (consult `data-source-registry.draft.md`).
2. Either correct the chain (likely the source step's tier was wrong) OR mark the data point with `confidence: Low` and review_notes explaining the gap.

The integrity of the chain map depends on this invariant.
