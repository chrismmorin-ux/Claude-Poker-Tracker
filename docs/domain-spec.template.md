## How to review

For each rule below, verify:

1. **Statement is correct** — does this rule actually hold in your domain, or did the engine misread the code?
2. **Source evidence is current** — does the cited file:line still implement the rule, or has the code moved/changed?
3. **Category is right** — is the rule in the correct problem-class bucket? (Categories map to `prog-domain-correctness.yaml` `problem_classes`.)
4. **Confidence is calibrated** — does High/Medium/Low match your read?
5. **Review notes addressed** — for any rule with non-empty review notes, the founder must confirm before promotion.

If a rule is wrong: edit the statement, downgrade confidence, or delete the rule entirely. If the engine missed a rule: add it under the appropriate bucket (use the same field structure).
