# Project: Player Tendencies Analytics

## Use Case

Between hands at a live table, the user checks their phone to review opponent tendencies before the next hand. Looking at a phone during a hand is not allowed, so the value is in **preparation and recall** — not real-time HUD display.

The primary surface is the **Players View**, where the user can quickly scan all opponents' calculated stats, inferred ranges, and custom exploit notes.

---

## Architecture

Three analytical layers, all built on a shared **Hand Timeline** abstraction:

```
  ┌─────────────────────────────────────────────────────┐
  │                  Player Profile                      │
  │                                                      │
  │  Tendencies          Ranges           Exploits       │
  │  (what they do)   (what they hold)  (what to do)     │
  │       │                  │                │           │
  │       └──────┬───────────┘                │           │
  │              ▼                             │           │
  │        Stat Engine ◄──────────────────────┘           │
  │              │                                        │
  │              ▼                                        │
  │       Hand Timeline                                   │
  │    (ordered action log)                               │
  │              │                                        │
  │     ┌────────┴────────┐                               │
  │     ▼                 ▼                               │
  │ actionSequence    seatActions                         │
  │ (new hands)      (old hands,                          │
  │                   best-effort)                        │
  └─────────────────────────────────────────────────────┘
```

### Layer 1: Hand Timeline
Normalized, ordered view of all actions in a hand. Single abstraction consumed by all analytics.

```js
// TimelineEntry
{ order: 1, seat: '3', action: 'raise', street: 'preflop' }
```

Built from `actionSequence` (precise, new hands) or reconstructed from `seatActions` (approximate, old hands). Every stat calculator and the future range engine consume timelines — never raw seatActions.

### Layer 2: Stat Engine
Consumes timelines. Answers ordered questions:
- "Did this player raise after another raise?" (3-bet)
- "Who was the last preflop raiser?" (C-bet aggressor)
- "What does this player do facing a flop bet?" (tendency buckets)

### Layer 3: Range Engine (future)
Consumes timelines + showdown cards (`allPlayerCards`). Builds:
- "After action line X, player showed Y" observations
- Inferred opening/continuing/folding ranges from showdowns + tendencies
- Range-aware exploit suggestions

## Phase 2: Persist actionSequence + Hand Timeline [COMPLETE]

### Goal
1. Persist `actionSequence` in saved hand records (trivial — it already exists in gameState, just not included in the save payload)
2. Build the `HandTimeline` abstraction that normalizes both new (actionSequence) and old (seatActions-only) hand data into a single ordered format

### Why This First
3-bet% and C-bet% require knowing action ORDER across seats. Without the timeline, we'd write heuristics that we'd immediately replace. The timeline is also the foundation the future range engine needs.

### Task 2a: Persist actionSequence

**File:** `src/hooks/usePersistence.js`

Add `actionSequence: gameState.actionSequence` to the handData object in both save paths (auto-save and saveNow). This is a 2-line change.

Old hands won't have actionSequence — the timeline builder handles this gracefully.

### Task 2b: Build Hand Timeline

**File (create):** `src/utils/handTimeline.js`

```js
// Build an ordered timeline from a hand record.
// Uses actionSequence if available (precise), falls back to seatActions (approximate).
buildTimeline(hand) -> TimelineEntry[]

// Get a specific player's actions in order from a timeline
getPlayerTimeline(timeline, seat) -> TimelineEntry[]

// Get all preflop actions in order
getPreflopTimeline(timeline) -> TimelineEntry[]

// Detect if a player faced a raise (another seat raised before them)
didPlayerFaceRaise(timeline, seat, street) -> boolean

// Find the last raiser on a street (for C-bet aggressor detection)
findLastRaiser(timeline, street) -> seat | null

// Check if a player bet the flop as PF aggressor
didPlayerCbet(timeline, seat) -> boolean
```

### Task 2c: seatActions fallback reconstruction

For old hands without actionSequence, reconstruct approximate order:
1. Group by street (preflop first, then flop, turn, river)
2. Within a street, iterate seats in position order (relative to dealer button)
3. Interleave each seat's actions in positional order

This is approximate (we don't know exact intra-street ordering) but gives a reasonable best-effort timeline for old data.

### Task 2d: Refactor tendencyCalculations to use timeline

Update `extractPreflopStats` and `extractPostflopAggression` to consume timelines instead of raw seatActions. The public API stays the same — `buildPlayerStats(playerId, hands)` — but internally it builds a timeline per hand first.

## Phase 3: 3-bet%, C-bet% via Timeline

### Goal
Add 3-bet% and C-bet% to the stat engine, using the timeline for precise ordered detection.

### Formulas
```
3-bet% = re-raised preflop / faced a raise preflop
  - Uses didPlayerFaceRaise(timeline, seat, 'preflop')
  - "Re-raised" = player raised on preflop AND they faced a prior raise

C-bet% = bet the flop as PF aggressor / saw flop as PF aggressor
  - Uses findLastRaiser(timeline, 'preflop') to identify aggressor
  - "Saw flop" = player has flop actions in timeline
  - "C-bet" = PF aggressor's flop actions include 'bet'
```

### Files
- **Modify:** `src/utils/tendencyCalculations.js` — add 3-bet/C-bet to accumulateHand
- **Modify:** `src/utils/tendencyCalculations.js` — add to derivePercentages
- **Modify:** `src/utils/__tests__/tendencyCalculations.test.js` — new tests

### Verification
- [ ] 3-bet: player raises after facing a raise = counted
- [ ] 3-bet: player opens (first raise) = not counted
- [ ] 3-bet: player who doesn't face a raise = not counted (denominator excluded)
- [ ] C-bet: PF aggressor bets flop = counted
- [ ] C-bet: PF aggressor checks flop = missed (counted in denominator only)
- [ ] C-bet: Non-aggressor bets flop = not a C-bet
- [ ] C-bet: PF aggressor folds preflop (no flop) = excluded from denominator

## Phase 6: Auto-Classification + Exploit Suggestions

### Goal
System suggests exploits based on stat thresholds.

### Suggestions
```
VPIP > 40%:    "Plays too many hands — tighten value range vs them"
VPIP < 12%:    "Very tight — steal blinds aggressively"
PFR < 5%:      "Rarely raises — strong hand when they raise"
AF < 0.5:      "Very passive postflop — bet for thin value"
AF > 4.0:      "Hyper-aggressive — call down lighter"
3-bet% < 3%:   "Rarely 3-bets — fold to their 3-bets"
3-bet% > 12%:  "Light 3-bettor — 4-bet bluff or call wider"
C-bet% > 80%:  "C-bets too much — check-raise flops"
C-bet% < 30%:  "Rarely C-bets — their C-bet is strong"
```

User can Accept (→ editable exploit), Dismiss (permanent), or Ignore.

## Future: Range Engine

Depends on: Phases 1-3 (timeline + stats), showdown data accumulation.

### Concept
When a hand reaches showdown and cards are revealed (`allPlayerCards`), record:
- What the player held
- Their full action line (from timeline)
- Their table position (from seat + dealer button)

Over time, build per-player observations:
- "From CO, opened and showed AJo, KQs, 77" → inferred opening range
- "Called 3-bet and showed AQo, JJ" → 3-bet calling range
- "Check-raised flop and showed middle set, flush draw" → check-raise range

Combined with tendencies (VPIP 35%, PFR 8% = loose-passive), the range engine can suggest:
- "This player's limp range is very wide (35% VPIP, 8% PFR = 27% limp rate)"
- "They've shown down 3 hands from EP: all premium (AA, KK, QQ) — tight opener"
- "Postflop: only bets with strong hands (AF 0.4), so their bet = strength"

### Data Flow
```
Hand Timeline → Showdown Observation → Range Profile
                      ↑                      ↓
                 allPlayerCards         Exploit Suggestions
                                       (range-aware)
```

This is a separate project but shares the timeline foundation built in Phase 2.

## Performance Considerations

- Calculate on hand save, not view render — store running totals in `player.stats`
- Incremental updates — one hand at a time, not full history re-scan
- Full recalculation only on first load or data repair
- Lazy calculation in `usePlayerTendencies` — only when Players View active
- Timeline is ephemeral (built per-hand, not stored)
